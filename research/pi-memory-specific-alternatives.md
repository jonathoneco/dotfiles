# Research: Pi memory/context package alternatives

## Summary

Best public-adoption signals currently point to **@samfp/pi-memory**, **pi-hermes-memory**, and the **pi-total-recall** meta-stack; **pi-context**, **pi-context-prune**, and **pi-observational-memory** are also popular but solve context/pruning/compaction more than durable memory. I found very little independent user-review evidence on Reddit/HN for the Pi-specific packages, so the ranking below leans on npm downloads, GitHub stars/forks/issues, Pi catalog presence, docs quality, and caveats.

Downloads below are npm `last-week` counts for **2026-05-10..2026-05-16**.

## Findings

1. **Most defensible picks by adoption are `@samfp/pi-memory` and `pi-hermes-memory`.** They have the highest weekly npm download counts among direct memory packages and reasonably active GitHub repos. `@samfp/pi-memory` is simpler and has an explicit scope warning; `pi-hermes-memory` is broader, adding searchable session history, skills, secret scanning, and failure memory. [@samfp npm API](https://api.npmjs.org/downloads/point/last-week/%40samfp%2Fpi-memory), [Hermes npm API](https://api.npmjs.org/downloads/point/last-week/pi-hermes-memory)
2. **`pi-total-recall` may be the best “recommended stack” if you want more than memory.** It bundles memory, session search, and knowledge search, and the `@samfp/pi-memory` docs recommend it as the complete context stack. [pi-total-recall](https://github.com/samfoy/pi-total-recall/), [@samfp/pi-memory Pi page](https://pi.dev/packages/@samfp/pi-memory)
3. **Context packages are popular but answer a different need.** `pi-context`, `pi-context-prune`, and `pi-observational-memory` have strong stars/downloads; use them for context window management, pruning, or compaction quality, not necessarily durable cross-session knowledge. [pi-context](https://pi.dev/packages/pi-context), [pi-context-prune](https://github.com/championswimmer/pi-context-prune), [pi-observational-memory](https://github.com/elpapi42/pi-observational-memory)
4. **MemPalace has external attention, but Pi-specific wrappers are not clearly winner-take-all.** HN has discussion of MemPalace itself, including both interest and skepticism, but the Pi wrappers (`mempalace-pi`, `@sinamtz/pi-mempalace`, `pi-mempalace`, `pi-mempalace-extension`) have low and split package signals. [HN MemPalace discussion](https://news.ycombinator.com/item?id=47672792), [mempalace-pi npm](https://registry.npmjs.org/mempalace-pi), [@sinamtz/pi-mempalace](https://registry.npmjs.org/@sinamtz/pi-mempalace)
5. **`pi-memctx` is worth a hands-on trial despite weaker adoption metrics.** It has the best published benchmark-style docs among the smaller packages, claiming large reductions in tool calls/tokens and better quality on local fixtures, but those are author-provided rather than independent. [pi-memctx Pi page](https://pi.dev/packages/pi-memctx)

## Name-collision caveats

- **`pi-memory` is overloaded.** `@samfp/pi-memory` is not the same as unscoped `pi-memory` (jayzeng). There are also GitHub repos named `tickernelz/pi-memory`, `becksclair/pi-memory`, `SiliconState/Pi-Memory`, and embedded extension copies. The `@samfp` docs explicitly warn to keep the scope. [@samfp Pi page](https://pi.dev/packages/@samfp/pi-memory)
- **`pi-mem` is overloaded.** Unscoped npm `pi-mem` points to georgebashi; search results also surface `jo-inc/pi-mem`, a plain-Markdown project with more GitHub stars, plus blog posts using the same name. [georgebashi/pi-mem](https://github.com/georgebashi/pi-mem), [jo-inc/pi-mem](https://github.com/jo-inc/pi-mem), [blog](https://pradeep.md/2026/02/11/pi-mem.html)
- **MemPalace naming is especially risky.** There is official MemPalace (`MemPalace/mempalace`, PyPI, official docs) plus several Pi wrappers and ports. The official MemPalace README warns about impostor domains, specifically including `mempalace.tech`; verify source before installing. [Official MemPalace](https://github.com/MemPalace/mempalace?tab=readme-ov-file)
- **`pi-supermemory` has scoped and unscoped variants.** The Pi catalog result found `@ramarivera/pi-supermemory`; an unscoped/forked `pi-supermemory` also appears with much lower npm usage. [@ramarivera Pi page](https://pi.dev/packages/@ramarivera/pi-supermemory), [cschmatzler/pi-supermemory](https://github.com/cschmatzler/pi-supermemory)
- **Context package names collide too.** `pi-context`, `pi-context-prune`, `pi-context-pruning`, and dynamic-context-pruning/DCP packages are related in theme but distinct in behavior.

## Gaps

- No solid independent user-review corpus was found for these Pi-specific memory packages. Treat stars/downloads as weak proxies, not proof of satisfaction.
- npm download counts can be skewed by automated installs and release workflows; GitHub stars can reflect hype or upstream monorepos, not Pi-specific use.
- Best next step: install 2–3 finalists (`@samfp/pi-memory` or `pi-total-recall`, `pi-hermes-memory`, and `pi-memctx` or `pi-memory-md`) in a throwaway Pi profile and compare latency, token use, memory quality, inspectability, and failure modes on the same tasks.
