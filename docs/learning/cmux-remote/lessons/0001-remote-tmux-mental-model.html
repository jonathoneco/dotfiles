Remote tmux is the server; cmux is the view

The remote tmux server owns processes and persistence. cmux connects with tmux control mode and renders that state using native macOS UI.

remote tmux session→cmux workspace tmux window→cmux tab tmux pane→cmux native split

One-time setup

  1. Enable Remote tmux in cmux Nightly's beta settings.
  2. Put host details in ~/.ssh/config. cmux honors aliases, ports, identities, and ProxyJump.
  3. Install tmux on the remote host and create named sessions there.
# On the remote host
tmux new-session -d -s wrangle -c ~/src/wrangle

# On the Mac
cmux ssh-tmux garden-pad

Daily rule

Create, rename, and destroy durable work in tmux terms. Use cmux to navigate and arrange the mirrored UI. Quitting cmux or losing SSH leaves the remote tmux server and its processes running.

Retrieval check: if a Codex process must survive your Mac rebooting, which object must own it: a cmux workspace or a remote tmux session?

First practical run

  1. Confirm ssh garden-pad connects.
  2. Create one disposable remote tmux session with two windows and one split.
  3. Run cmux ssh-tmux garden-pad locally.
  4. Verify the 1:1 mapping above, then disconnect cmux and confirm the remote processes survive.
