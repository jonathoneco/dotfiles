# Workflow Plan

Enter the planning phase of a workflow. Synthesizes research findings (and optionally an external document) into a structured architecture document that becomes the foundation for specs.

## Arguments

- `<name>` — the workflow slug (kebab-case)
- `[external-doc-path]` — optional path to an external document (Notion export, PRD, RFC, etc.) to use as primary input alongside research findings

## Process

### Step 1: Load context

Read these files to orient (parallel reads):

1. `.workflows/<name>/research/handoff-prompt.md` — **primary context**. This was generated by the research phase session with deep context from doing the work. Trust its instructions.
2. `.workflows/<name>/research/index.md` — summaries only. Do **not** read raw research notes unless the handoff prompt specifically directs you to a particular note.
3. `.workflows/<name>/state.json` — verify `current_phase` is `plan` (or that research is completed)
4. If `[external-doc-path]` is provided, read that document.

If `current_phase` is not `plan` and research phase is not `completed`, report the mismatch and suggest checking `/workflow-status <name>`.

### Step 2: Set phase active

```bash
bd set-state <epic-id> phase=plan
```

Update `state.json`:
- `phases.plan.status` = `"active"`
- `phases.plan.started_at` = current ISO 8601 timestamp (only if first activation)
- `current_phase` = `"plan"`
- `updated_at` = current ISO 8601 timestamp

### Step 3: Present and Ask

Summarize the loaded context for the user:

- **Handoff prompt highlights**: Key findings and instructions from the research phase
- **Research summary**: Main topics investigated, conclusions reached
- **Key decisions**: Technology choices, pattern decisions, dead ends from research
- **Component Map seed**: If the handoff prompt suggests components, present them

Then ask for direction:

"Before I draft the architecture, what's your vision for the high-level approach? Any constraints, priorities, or technology preferences I should know?"

**Do NOT begin writing architecture.md until the user provides direction or confirms the approach.**

### Step 4: Select planning mode

**Mode A — Synthesize from research**: When no external doc is provided, distill research findings into architecture. The handoff prompt and research index are your primary inputs.

**Mode B — Ingest external document**: When `[external-doc-path]` is provided:

1. Read the external document in full
2. Write `.workflows/<name>/plan/external-ref.md` with:
   - Link to the original document
   - One-paragraph summary
   - Key requirements extracted
3. Use the external doc as primary input for architecture, supplemented and validated by research findings

### Step 5: Produce architecture document

Write `docs/feature/<name>/architecture.md` following the proven pattern:

```markdown
# <Title> — Architecture

| Field | Value |
|-------|-------|
| Status | Draft |
| Workflow | <name> |
| Created | <date> |

## Overview
[What this feature/system does, why it exists, who it serves]

## System Context
[How it fits into the existing gaucho architecture — reference existing patterns
in cmd/server/main.go, internal/ structure. Be specific about where new code
integrates with existing code.]

## Tech Stack Decisions
[Technology choices with rationale — reference specific research findings.
Include what was considered and rejected, with pointers to dead-ends.md.]

## Architectural Patterns
[Key patterns to follow: handler -> service -> database layering, HTMX partial
rendering, constructor injection, slog logging, etc. Call out any NEW patterns
this feature introduces and why.]

## Data Model
[New tables, relationships, migrations needed. Include column types and
constraints. Reference existing tables that will be joined or extended.]

## Data Flow
[Request flow through the system — from HTTP request to response. Cover
both full-page loads and HTMX partial updates.]

## Component Map
[High-level breakdown of implementation areas — these become specs in the
next phase. Each component should be a cohesive unit of work with clear
boundaries.]

## Phasing Summary
[Rough implementation order — foundation -> core -> integration -> polish.
Identify what must be done first and what can run in parallel.]

## Open Questions
[Decisions deferred to spec phase. Be explicit about what information
is needed to resolve each question.]
```

Ground every section in specific file paths, existing patterns, and research findings. Avoid generic architecture prose — this document should be actionable for the spec phase.

### Step 6: Review gate

Present the architecture doc outline to the user — section headings with one-line summaries for each.

Ask: "Here's the architecture outline. Want to review it before I finalize, or should I continue to the handoff prompt?"

**Do not** proceed to handoff generation without user acknowledgment.

### Step 7: Generate handoff prompt

Write `.workflows/<name>/plan/handoff-prompt.md`:

```markdown
# Handoff: plan -> spec

## What This Phase Produced
- Architecture document at `docs/feature/<name>/architecture.md`
- [Any other artifacts]

## Key Artifacts
| File | Purpose |
|------|---------|
| `docs/feature/<name>/architecture.md` | Full architecture with component map |
| `.workflows/<name>/plan/external-ref.md` | External doc summary (if applicable) |

## Component Map (becomes specs)
[List each component from the architecture doc — these are the spec topics]
1. <component> — <one-line description>
2. <component> — <one-line description>

## Tech Stack Decisions
[Summarize key technology choices so the spec phase has them at hand]

## Open Questions for Spec Phase
[Questions deferred from planning that specs must address]

## Instructions for Spec Phase
1. Read `docs/feature/<name>/architecture.md` first
2. Create cross-cutting contracts (00-cross-cutting-contracts.md)
3. Write one numbered spec per component from the Component Map
4. Each spec must include: existing code context, files to create/modify, acceptance criteria
5. Establish dependency ordering between specs
```

### Step 8: Resolve research gate

If a gate issue was created when the research phase ended, resolve it:

```bash
bd close <gate-id> --reason="Plan phase complete — architecture document written"
```

Where `<gate-id>` comes from `state.json` at `phases.research.gate_id`.

### Step 9: On phase end

Suggest running `/workflow-checkpoint --phase-end` to finalize and transition to the spec phase. The checkpoint will:
- Mark the plan phase as completed
- Generate the handoff prompt (if not already written in Step 5)
- Create a gate issue for review before spec begins
- Advance `current_phase` to `spec`

## Key principles

- **The handoff prompt is primary.** The research phase session had deep context. Trust its handoff prompt over your own interpretation of raw research notes.
- **Architecture drives specs.** The Component Map in the architecture document directly becomes the list of specs. Every component should be clearly bounded and cohesive.
- **Reference, don't repeat.** Point to research notes and dead-ends by path rather than copying their content into the architecture document. Keep the architecture doc focused on decisions and structure.
- **External docs supplement, not replace.** When ingesting a PRD or RFC, validate it against research findings. Flag conflicts between the external doc and what the research phase discovered.
- **Actionable over aspirational.** Every section should contain specific file paths, interface shapes, or migration schemas. If a section would be pure prose with no concrete details, it belongs in Open Questions instead.
- **Architecture decisions are human decisions.** Present options and trade-offs. Never pick a technology, pattern, or data model without the user's input.
